# Channel-Based Diarization Support

## Context

AudioCaptureKit captures mic (local speaker) and system audio (remote participants) but currently **mixes the mic into both stereo channels** (`Left = mic + system_L`, `Right = mic + system_R`). This makes downstream speaker diarization difficult: to run independent processing per speaker, you need access to clean channel-separated audio.

This feature adds:
1. A new delegate callback exposing **raw, full-quality per-channel buffers** (mic mono + system stereo) before mixing â€” callers decide how to process them
2. A configurable `MixingStrategy` so callers can choose the WAV channel layout
3. Explicit channel/layout metadata in `RecordingResult`
4. Optional raw PCM sidecar files (mic mono + system stereo) for use with external tools

**Design principle**: AudioCaptureKit provides maximum-quality raw data. It is the caller's responsibility to fold, downsample, or otherwise optimize for a specific use case (e.g. ASR). The library never degrades audio quality on behalf of the caller.

The `MixingStrategy` change affects the output WAV layout (the only breaking change, gated behind configuration). All other changes are purely additive. Both macOS Swift and Windows Rust must stay in sync.

## Step 2: `MixingStrategy` â€” Configurable WAV Channel Layout

### New enum

**Swift** â€” new file `macOS/Sources/AudioCaptureKit/Models/MixingStrategy.swift`:

```swift
/// Controls how microphone and system audio are combined into the output WAV file.
///
/// Choose the strategy that best fits your downstream use case. The library
/// always captures at full quality; this setting only affects how samples are
/// written to disk. Raw per-channel samples (before mixing) are always available
/// via ``AudioCaptureDelegate/captureSession(_:didProduceChannelBuffers:)``.
public enum MixingStrategy: Sendable, Codable {
    /// Mic is mixed into both stereo channels alongside system audio.
    /// - Left:  mic + system_L
    /// - Right: mic + system_R
    /// This is the default and matches the original AudioCaptureKit behavior.
    case blended

    /// Mic and system audio occupy separate stereo channels with no blending
    /// of mic into system or vice versa.
    /// - Left (Ch 1):  mic only (mono mic signal)
    /// - Right (Ch 2): system audio, mono-folded as (system_L + system_R) / 2
    ///
    /// The mono-fold on the right channel preserves audio from both the left
    /// and right system channels â€” no content is discarded. If you need full
    /// stereo system audio in the WAV file, use a 3-channel configuration
    /// (channels: 3, future support).
    case separated

    /// Reserved for future 3-channel multi-mic configurations.
    /// Currently behaves identically to ``separated``.
    case multichannel
}
```

## Step 3: Update `StereoMixer`

### macOS: `macOS/Sources/AudioCaptureKit/Processing/StereoMixer.swift`

**Deprecate** (do not delete) `mixMicWithStereoSystem`:
```swift
@available(*, deprecated, renamed: "mix(mic:system:strategy:)")
public func mixMicWithStereoSystem(...) -> [Float]
```

**Add** a unified `mix(mic:system:strategy:) -> [Float]` method:
```swift
/// Mixes mic and system audio according to the specified strategy.
/// - Parameters:
///   - mic: Mono mic samples, resampled to target rate.
///   - system: Interleaved stereo system samples [L0, R0, L1, R1, â€¦].
///   - strategy: How to combine the sources.
public func mix(mic: [Float], system: [Float], strategy: MixingStrategy) -> [Float] {
    switch strategy {
    case .blended, .multichannel:
        return mixMicWithStereoSystem(mic: mic, system: system)
    case .separated:
        return separateChannels(mic: mic, system: system)
    }
}
```

**Add** `separateChannels(mic:system:) -> [Float]` (used by `.separated` strategy):
- Left  = mic[i] (zero-padded when shorter)
- Right = (system[2*i] + system[2*i+1]) / 2  â€” mono-fold preserving both L and R channels
- Frame count = max(mic.count, system.count/2)

This is the correct way to fit stereo system audio into a single channel without discarding half the content (e.g., audio panned right would be lost if only system_L were used).

No `foldToMono` is exposed as a standalone public API. Callers who need system mono from `ChannelBuffers.systemSamples` can compute `(system[2i] + system[2i+1]) / 2` themselves.

### Windows: `windows/audio-capture-core/src/processing/stereo_mixer.rs`

Mirror the same changes: deprecate `mix_mic_with_stereo_system`, add `mix()` dispatching on `MixingStrategy`, add `separate_channels`.

## Step 5: Add New Model Types

### `ChannelBuffers` â€” full-quality, unprocessed per-channel samples

**Swift** â€” add to `macOS/Sources/AudioCaptureKit/Models/AudioModels.swift`:
```swift
/// Raw per-channel audio samples from one processing cycle (~100 ms window).
///
/// Delivered via ``AudioCaptureDelegate/captureSession(_:didProduceChannelBuffers:)``
/// before mixing and file writing. All samples are Float32, normalized to [-1.0, 1.0],
/// at the session's configured sample rate.
///
/// ``systemSamples`` is full interleaved stereo [L0, R0, L1, R1, â€¦]. The library
/// does not fold, downsample, or otherwise modify system audio. If your use case
/// requires mono system audio (e.g. for single-channel ASR), average the two channels
/// yourself: `(system[2*i] + system[2*i+1]) / 2`.
public struct ChannelBuffers: Sendable {
    /// Mono microphone samples. Empty when mic capture is disabled.
    public let micSamples: [Float]

    /// Interleaved stereo system audio [L0, R0, L1, R1, â€¦].
    /// Empty when system capture is disabled.
    public let systemSamples: [Float]

    /// Sample rate of both arrays, in Hz.
    public let sampleRate: Double

    /// Wall-clock timestamp at the start of this buffer window.
    public let timestamp: Date
}
```

**Rust** â€” add to `windows/audio-capture-core/src/models/audio_models.rs`:
